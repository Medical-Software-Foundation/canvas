## Detailed Findings

### Plugin API Server Security

`NoteProductionDashboardAPI` at `note_production_dashboard/handlers/dashboard_api.py:193` inherits `(StaffSessionAuthMixin, SimpleAPI)`. All three routes (`/dashboard`, `/providers`, `/providers/<provider_id>/notes`) are gated by the mixin — no manual `authenticate()` to audit, which is the recommended pattern.

**Authentication & authorization**

- `StaffSessionAuthMixin` enforces a logged-in staff session for every request. Any non-staff session — including unauthenticated requests, patient-portal sessions, and stale cookies — is rejected before the route body runs.
- The application is registered with `scope: "provider_menu_item"` in the manifest, so the entry point is also staff-only. Mixin and scope are aligned.
- `provider_id` from the URL path (`/providers/<provider_id>/notes`) is used solely as a grouping key on `note__provider__id` (`dashboard_api.py:185`). It is not a tenant identifier, and per the plugin spec all staff are authorized to see all locked notes; there is no per-staff filtering gap to exploit. Django ORM parameterizes the filter, so the path param cannot be used for SQL injection.

**Input validation**

- `period` is coerced to `"daily"` if not in `("daily", "weekly", "monthly")` at `dashboard_api.py:202–203`, `:214–215`, and `:255–256`.
- `week_start` is coerced to `"sunday"` if not in `("sunday", "monday")` at `dashboard_api.py:204–205`, `:216–217`, and `:257–258`.
- These coercions happen on every endpoint *before* the values are used to compute time windows or rendered into the HTML response.

**XSS in HTML response (`/dashboard`)**

- The dashboard HTML is built via Python f-string with three substitutions: `{period}`, `{week_start}`, and `{cache_bust}`.
- `period` and `week_start` are constrained to fixed enums above, so they cannot carry an XSS payload.
- `cache_bust` is a module-level constant computed from `datetime.now()` at module load (`dashboard_api.py:35`); it is never user-controlled.
- All `innerHTML` writes in the embedded JS use **static** string literals (`dashboard_api.py:596, 598, 610–611, 615, 643, 668, 684, 712, 768`). Dynamic note/provider data is rendered exclusively via `td.textContent = …` and `nameSpan.textContent = …`, which DOM-escapes by construction.
- Provider and patient names — which can be arbitrary user-entered strings — never reach `innerHTML`.

**JSON responses (`/providers`, `/providers/<id>/notes`)**

- Returned via `JSONResponse(...)`, which serializes via the SDK's encoder. No user-controlled string interpolation into the response body.

**Logging**

- One `log.info` call at `dashboard_api.py:220` reports `period`, `week_start`, and the computed window edges. No patient names, no note IDs, no PHI, no secrets.
- Provider/patient details are returned in the JSON body but never written to logs.

**HTTP method exposure**

- All three routes are `@api.get(...)` only. No state-changing operations, so CSRF risk is not applicable; staff-session auth on a same-origin GET is sufficient.

### FHIR API Client Security

### Application Scope

| Manifest field | Value | Aligned? |
|---|---|---|
| `applications[0].scope` | `provider_menu_item` | yes |
| API auth mixin | `StaffSessionAuthMixin` | matches scope |

A `provider_menu_item` application is launched from the Canvas provider sidebar, which is staff-only by definition. Pairing it with `StaffSessionAuthMixin` on the backing API enforces the same audience at both surfaces.

By spec, the dashboard intentionally has no role gate — any logged-in staff member (clinician, MA, receptionist, admin) can view all providers' note counts. This is an explicit product decision, not a security defect. If team-based restriction is later required, it would need a manual `authenticate()` override that consults `TeamMembership`.
